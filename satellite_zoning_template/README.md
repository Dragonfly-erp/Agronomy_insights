# Шаблон: супутникові знімки → зони + рельєф

Багаторазовий конвеєр: із супутникового **NDVI** одного поля робить **рівно два
шейпфайли** — зони продуктивності та рельєф. Розроблено на прикладі поля П1
(Софіївка), придатне для будь-якого поля.

![приклад](example_P1.png)

## Вихід (рівно 2 файли)
| Файл | Що | Атрибути |
|------|----|----------|
| **`zones.shp`** | зони продуктивності (полігони), покривають **повну площу поля** до країв, **суцільне покриття без щілин** | `zone_id` (1=найслабша…N=найсильніша), **`product`** (продуктивність зони, поле=100), `veg_min`, `veg_max`, `veg_mean` (NDVI), `area_ha`, `corr_elev` (кореляція NDVI↔висота в зоні) |
| **`relief.shp`** | рельєф поля — **точки висот** (для Ag Leader / 3D-поверхні) | `elev_m` (висота, м), `slope_pct` (крутість, %). Крок сітки — `POINT_STEP_M` (20 м). Альтернативи: `RELIEF_TYPE="contours"` (горизонталі) або `"elevation_zones"` |

> **`product`** — відносна продуктивність зони: середній NDVI зони / середній NDVI
> поля × 100 (поле = 100). Показує, наскільки зона сильніша/слабша за середнє.
> Через насичення NDVI при повному покриві розкид помірний, але **порядок зон
> надійний** (підтверджено рельєфом).

Поряд кладуться `.qml` (стилі QGIS) — відкрив `.shp`, одразу бачиш кольори.

## Як користуватись
1. Відкрий `run_zoning.py`, у блоці **CONFIG** впиши шляхи:
   ```python
   INPUT_NDVI    = "шлях/до/ndvi.shp"   # NDVI-полігони (атрибут NDVI)
   FIELD_CONTOUR = "шлях/до/field.shp"  # контур поля; None = взяти обвід NDVI
   OUT_DIR       = "output"
   N_ZONES       = 5                    # 4..6
   SMOOTH_LEVEL  = "C"                  # A точно · C рекомендовано · D сильно згладжено
   RELIEF_TYPE   = "points"             # points (Ag Leader/3D) · contours · elevation_zones
   POINT_STEP_M  = 20                   # крок сітки точок висот, м
   ```
2. Запусти:
   ```bash
   pip install geopandas rasterio scikit-learn scipy shapely
   python run_zoning.py
   ```
3. Забери `OUT_DIR/zones.shp` і `OUT_DIR/relief.shp`.

## Що робить конвеєр (стисло)
1. читає NDVI, переводить у метричну проєкцію UTM (визначає зону автоматично);
2. будує контур поля (з `FIELD_CONTOUR` або з обводу NDVI);
3. **аналіз на «продуктивному ядрі»** (поле, стягнуте на `EDGE_BUFFER_M`=18 м) —
   так відкидаються крайовий ефект, розворотні смуги, узбіччя, проходи техніки
   (медіанний фільтр) та викиди (MAD);
4. кластеризація за малюнком NDVI → впорядкування зон за силою вегетації →
   прибирання тонких «відростків» і злиття шматків < `MIN_FRAC` площі
   (компактні, придатні для техніки зони);
5. **розтягування зон на повний контур поля** — кожен крайовий піксель дістає
   найближчу зону, тож `zones.shp` покриває всю площу, яку ви дали;
6. рельєф — з безкоштовної ЦМР **Copernicus GLO-30** (ESA, ~30 м) за контуром
   (автозавантаження; або вкажіть локальний DEM у `DEM_PATH`).

## Багаторічні зони + пакетна обробка
`INPUT_NDVI` може бути **списком знімків** одного поля за різні роки — тоді кожен
рік нормалізується (поле=1) і вони усереднюються у **стабільний патерн
продуктивності** (це надійніше, ніж один рік). `veg_*` — реальний NDVI (сер. по
роках), `product`/`corr_elev` — по стабільному патерну.

Для **багатьох полів** — `run_batch.py`: сканує теку з NDVI-шейпфайлами, групує за
полем, сам відбирає **канопі-роки** (роки з голим ґрунтом відкидає), і для кожного
поля кладе `zones.shp` + `relief.shp` у `output_batch/<Поле>/`. Якщо у поля немає
жодного канопі-року — береться найкращий наявний (позначається як низька
впевненість). Налаштування — вгорі `run_batch.py` (`INPUT_DIR`, `CANOPY_MIN`).

## 🛰️ Професійний режим — власні Sentinel-2 + PDF-звіт (`zoning_pro.py`)
Найякісніший варіант: **не потрібен готовий NDVI-експорт** — модуль сам тягне
безхмарні знімки **Sentinel-2 L2A** (10 м, ESA/Copernicus, безкоштовно, через
відкритий STAC Earth Search) за кілька років і рахує **три індекси**:
- **NDVI** — вигор рослин,
- **NDRE** — червоний край (не насичується у щільному покриві),
- **NDMI** — вологозабезпеченість.

З них будується **стабільний багаторічний медіанний композит** (голий-ґрунтові й
хмарні сцени відкидаються за SCL) → зони на кількох індексах.

Вихід на поле (`<FARM>/<FIELD>/`): `zones.shp` (атрибути `zone_id`, `product`,
`ndvi`, `ndre`, `ndmi`, `area_ha`, `corr_elev`) + `relief.shp` (точки висот) +
**`report.pdf`** — короткий звіт на 1 стор.: 6 карт (NDVI/NDRE/NDMI/рельєф/
знімок/зони), джерело й період даних, дати сцен, таблиця зон. Для показу клієнту.

```python
# zoning_pro.py — CONFIG:
FARM="ГЕВОРГ"; FIELD_NAME="P4"; BOUNDARY_SRC="поле.shp"  # лише контур поля
YEARS=range(2019,2025); MONTHS=(6,8)                     # роки й вікно піку
python zoning_pro.py
```
Багато полів господарства — `run_batch_pro.py` (тека контурів → `<FARM>/<Поле>/…`).

**Дані/джерела:** Sentinel-2 L2A — Element84 Earth Search (`earth-search.aws.element84.com`,
AWS Open Data, без ключа); рельєф — Copernicus GLO-30 (AWS Open Data). Потрібен інтернет.

## Цілісність покриття (гарантія)
`zones.shp` — це **суцільне покриття без щілин і перекриттів**. Це забезпечено
жорстко:
* межі зон спрощуються **топологічно-цілісно** (`shapely.coverage_simplify`) —
  спільні межі сусідніх зон лишаються збіжними (а НЕ спрощуються окремо, що
  колись відкривало «слівери»);
* після перепроєкції координати **прив'язуються до сітки** (`set_precision`);
* будь-яка залишкова щілина **детерміновано зливається** із сусідньою зоною
  (`fill_coverage_gaps`);
* нарешті файл **перечитується з диска й перевіряється** — якщо лишилась хоч
  одна щілина, скрипт **зупиняється й НЕ зберігає** биту версію
  (`gaps(holes)==0` — обов'язкова умова).

## Рівні згладжування
`A` — максимально точно (повторює малюнок NDVI) · `B` — легко · **`C` —
рекомендовано** (баланс точність/простота) · `D` — сильно (прості блоки).

## Приклад
`output_P1_example/` — готовий результат для поля П1 (86 га): `zones.shp` (5 зон,
повне поле) + `relief.shp` (горизонталі 1 м), у WGS84 (EPSG:4326).

## Примітки
- Вихідна CRS — `OUT_CRS` (за замовч. **WGS84**); площі рахуються у метричній
  UTM і зберігаються в `area_ha`.
- `corr_elev` — коефіцієнт Пірсона NDVI↔висота в межах зони: наскільки
  продуктивність зони пояснюється рельєфом (від'ємний = нижче за рельєфом ⇒
  сильніша вегетація — типово для цього поля).
- Для рельєфу потрібен інтернет (або локальний `DEM_PATH`).
